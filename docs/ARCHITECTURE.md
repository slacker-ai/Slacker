# Architecture

Engineering overview of how Slacker is built. For *what* to build see
`IMPLEMENTATION.md`.

## Principles

- **Local-first.** No backend, no telemetry. Network egress is limited to `slack.com`,
  the user's own LLM endpoint, and GitHub release assets for signed Sparkle updates.
- **BYO key.** LLM calls use the user's own provider/key (Keychain).
- **Precision over recall.** Uncertain items are never auto-surfaced; they go to the
  review queue. A noisy feed gets abandoned faster than a feed that misses things.
- **Item-centric, never per-person.** No view aggregates by employee.
- **Testable seams everywhere.** All I/O goes through injectable protocols so the
  pipeline is unit-tested without real network, processes, or disk.

## Stack

Swift 5.9+ / SwiftUI, macOS 14+. `MenuBarExtra` + a main `WindowGroup`. Swift
Concurrency (actors). **GRDB.swift** over SQLite (migrations from day one). Keychain
for secrets. `URLSession` + `Codable`. Project generated by **XcodeGen** from
`project.yml` (the `.xcodeproj` is gitignored). App Sandbox is **off** (so the CLI LLM
backends can spawn subprocesses); distributed as a notarized DMG and updated through
Sparkle's EdDSA-verified appcast.

## Runtime data flow

```
AppRoot (@MainActor)
  └─ SyncCoordinator (actor; no recurring timer)
       ├─ SocketModeClient per configured workspace
       │    ├─ apps.connections.open(xapp token) → temporary wss URL (memory only)
       │    ├─ acknowledge each envelope before application work
       │    ├─ route Events API payload by team_id
       │    └─ reconnect on failure/disconnect/refresh_requested with bounded jitter
       ├─ debounced event ingestion per workspace/channel
       │    ├─ new root          → conversations.history from lastPolledTS
       │    ├─ reply/edit        → conversations.replies for affected root
       │    ├─ reaction change   → resolve local message to root, then replies
       │    └─ deletion          → remove local message/item, refresh parent if needed
       ├─ lifecycle gap recovery (launch/wake/reconnect only)
       │    ├─ history from durable lastPolledTS
       │    └─ tracked thread snapshots in batches of 12 (max 3 HTTP requests)
       ├─ newly watched channel Task
       │    └─ channel-only initial history backfill + targeted root detection
       ├─ analysis path
       │    ├─ Socket batch → detect only affected roots
       │    ├─ lifecycle snapshots → detect only roots whose Slack content changed
       │    └─ update UI through GRDB observation
       └─ coalesced background enrichment
            ├─ ItemThreadSummaryService.analyzeOpenThreads(changed roots)
            └─ SummaryService.generateDailySummaries(changed channels)

UI: MainViewModel / OverviewViewModel observe the DB via GRDB ValueObservation and
re-render as realtime sync writes. Triage actions write Item changes + TriageLabels, and
fire PatternEvolutionService.evolveFromTriage (immediate automatic learning) in the
background.
```

Slack-generated `channel_join`, `channel_leave`, `group_join`, and `group_leave` message
subtypes are discarded during ingestion. A text-signature fallback removes rows created by
older builds before connected-user mention routing or calibration can consume them.

All Slack/detection work runs off the main actor; view models are `@MainActor`
`@Observable`. GRDB uses a `DatabasePool` (concurrent reads, serialized writes).

## Classification engine

Detection is a layered classifier, not one LLM pass. Each level is allowed to stop early
only when it has enough confidence; otherwise the item moves to the next level or to the
Review queue. This keeps the cheap, deterministic path fast while preserving the invariant
that uncertain items never appear in Needs attention.

**Level 0 — ingestion eligibility.** `IngestionService` stores Slack messages and threads
idempotently. `DetectionService` evaluates watched channels, roots newer than the
channel's `lastDetectedTS`, roots for active items, roots explicitly invalidated by edits,
and roots with new messages that
mention the connected user. That keeps restart cycles from reclassifying the whole archive
while still letting new thread replies update open work.

**Level 1 — deterministic rules.** `RuleEngine` maps message text to an intermediate
`MessageClass` with confidence:
- `openQuestion`: directed asks, review/approval/share requests, ping/page/loop-in
  handoffs, help requests, owner questions, deadline asks.
- `decisionPending`: approve/reject, ship/hold, go/no-go, rollback/roll-forward,
  pick-one, "should we" language.
- `blocker`: explicit blocked/waiting/dependency language, access/permission blockers,
  broken build/test/deploy pipelines, incidents, timeouts, elevated errors, degraded
  service, outages, release/deploy blockers.
- `contextOnly`: FYI, chatter, status updates, already-completed work, or empty text.

Rules are deliberately narrow. Explicit blockers win over questions, but explicit
coordination asks such as "ping/page/loop in on-call about timeouts" stay `openQuestion`
even though the topic contains incident words.
Before rule or LLM analysis, Slacker strips triple-backtick fenced blocks from message
text so pasted logs do not create false asks, blockers, or resolution signals.

**Level 2 — signal mapping and thresholds.** `Classifier` maps class → product signal:
- `openQuestion` → `missed_followup`
- `decisionPending` / `blocker` → `stale`
- direct tag of the connected user (`<@workspace.authUserID>`) → `mention`
- `contextOnly` → no item

It then applies thread-context discounts (for example, likely answered threads reduce
confidence) and routes by thresholds. Follow-up/bump/checking-in replies in a thread are
treated as stale because they indicate the loop is aging rather than closed. Defaults are:
- `confidence ≥ 0.80` → `surfaced` (Needs attention)
- `0.50 ≤ confidence < 0.80` → `review`
- `< 0.50` → no item

Channel sensitivity and calibration can shift these thresholds, but the rule remains:
**uncertain is never auto-surfaced.**

**Level 3 — LLM escalation.** `LLMClassifier` is called for review-band candidates and,
when approved learned guidance exists, as a precision check on candidates the rules would
surface. It runs only when the user has configured an LLM provider. The prompt returns strict JSON:
`{"class":"openQuestion|decisionPending|blocker|contextOnly","confidence":0.0-1.0}`.
Parse or provider failure keeps the rules verdict unchanged. Without approved guidance,
the LLM is not called for messages the rules confidently surface; confident discards never
incur an LLM call.

**Level 4 — persistence and idempotency.** `DetectionService` upserts items on
`(channelID, rootMessageTS)`. User-terminal states are not overwritten; legacy `snoozed`
rows are also left untouched for local database compatibility. A later stronger
classification may promote review → surfaced but does not revive dismissed work.

**Level 5 — resolution backstop.** Open items are continually checked by the resolution
layers below. This is separate from initial classification because many false positives are
items that were once actionable but have since been answered, handed off, or fixed.

## Resolution (two layers)

An open item auto-closes (and drops off the dashboard, which observes item changes live)
when **either** layer fires:

1. **Heuristic** (`ResolutionDetector`, no LLM, every cycle): resolved emoji/reactions
   (✅/☑/👍-style) anywhere in the thread, or explicit "done/fixed/shipped/resolved/…"
   wording anywhere in the thread. Newly observed done reactions are ordered by local
   observation time, so adding ✅ to an older thread message can close the current loop
   without letting old reactions re-close reopened work. Reaction ordering is explicit, so
   a newly-added ✅ also beats an older 👀 on the same message. Open/in-progress
   emoji/reactions (👀/⏳/⚠️-style) and same-message actionable asks keep the item open and
   prevent false closes like "not fixed yet 👀" or "can be done prior". A *bare* reply ("looking into it")
   deliberately does NOT close anything here. Coordination asks to ping/page/notify
   someone can close on concrete handoff replies like "paging now".
2. **LLM solution-judge** (`ItemThreadSummaryService`, when a thread grows): one LLM call
   returns `{summary, resolved, confidence}`. Auto-closes (`reason: inferred`) when
   `resolved` && `confidence ≥ 0.80` && `replies ≥ 1` — so a single reply that actually
   answers an open question closes it, while a non-solution reply stays open. If the latest
   message is actionable by deterministic rules, the item stays open even when the LLM says
   `resolved: true`. Approved learned guidance is appended to this analyzer too, so approved
   resolution guidance can affect future thread summaries/resolution. No extra cost beyond
   the summary it already generates per new batch of replies.

Resolved and dismissed items are still watched for new thread activity. New or newly
edited activity is ordered by durable local observation time, then checked by a narrow
compiled regex bank before the normal phrase rules and optional LLM guidance. Actionable
activity (for example, "this is still failing" or "can someone look?") reopens the existing
item, clears its `resolutionReason` and cached summary, and routes it through the same
surfaced/review thresholds. A new open reaction/emoji reopens immediately; removing the
final done reaction reopens an item that was closed by that reaction. Non-actionable
activity leaves the terminal decision intact. Legacy snoozed rows remain terminal.

Mentions are a separate surfacing path, keyed to the connected user's Slack id stored on
`Workspace.authUserID`. A mention creates a `mention` item only when the thread is not
already active. If a dismissed or resolved item receives a newer mention, it is revived as
`mention`; older mentions do not resurrect terminal rows.

## Calibration flywheel

Every triage (review verdict, dismissal, mark-resolved) writes a `TriageLabel`.
`CalibrationService` shifts a channel's surface/review thresholds by its ignore-rate once
≥5 labels exist — fitting one manager's definition of attention over time.

## Self-evolution (action-driven patterns)

Every Resolve, Dismiss, This matters, or Ignore action fires `evolveFromTriage` in the
background with the current root, replies, reactions, effective prompts, and recent
contrasting labels. The strict-JSON result can contain one validated channel phrase, a
global guidance addition, and a source-channel guidance addition. Useful output is written
as approved immediately; missing messages, provider failures, unsafe phrases, duplicates,
and parse failures write nothing and never block the user action.

The immutable built-in classifier and resolution contracts are followed by the editable
global overlay and the source-channel overlay. When an automatic append makes those learned
documents total at least 8,000 characters, a second model call condenses both documents.
The replacement is accepted only when it is shorter, below the threshold, and neither
document changed while the call was running. Settings exposes both editable documents and
read-only previews of the effective classifier and resolution prompts. Revert all restores
the built-in behavior. Historical proposed/rejected states remain only for migration.

## Data model (GRDB)

`appSettings` (single row: staleness, summary interval, LLM settings, manifest/team; the
legacy `pollIntervalSeconds` column remains decodable but is ignored) · `workspace`
(one Keychain-backed xoxp + xapp credential pair) · `channel` (watch flag + sensitivity +
durable HTTP `lastPolledTS` + `lastDetectedTS`) ·
`message` (PK `channelID:ts`, threadTS, reactions JSON, first-observed/edit/open-reaction/
resolved-reaction-added/removed observation times;
indexed on `(channelID,ts)` and
`threadTS`) · `item` (the unit of attention — type (`missed_followup` / `stale` /
`mention`), state machine, confidence,
resolutionReason, threadSummary) · `label` (calibration) · `summary` (PK
`channelID:date`) · `user` (grounding cache) · `learnedPattern` / `learnedGuidance`
(self-evolution, §7.5a; `evolutionRun` remains from the former batched pass but is now
unused). Migrations v1–v18, append-only.

**Item state machine:** `open → surfaced | review → resolved | dismissed`, with resolved
or dismissed returning to surfaced/review only after explicit new activity. Legacy local
databases may still contain `snoozed` rows; the current UI no longer creates them.
Only `surfaced` items appear in Needs attention and the badge count.

## LLM provider abstraction

One `LLMClient` protocol, seven backends behind `LLMClientFactory`:
- **HTTP/key:** OpenAI, Anthropic, Gemini, generic OpenAI-compatible, Ollama — all over
  the `HTTPTransport` seam.
- **CLI/subprocess:** Codex (`codex exec`), Claude Code (`claude -p` via stdin) — over
  the `CLIRunner` seam; binary located by override → PATH → common dirs.

Used for (a) ambiguous classification, (b) thread summary + resolution, (c) daily
summaries, and (d) learned-pattern/guidance proposals. Strict-JSON contracts with
defensive parsing; failures degrade to rules-only.

## Security & privacy

- Secrets only in Keychain (`KeychainStore`); never DB/plist/logs. User and app tokens are
  isolated under `slack.user.token.<workspaceID>` and `slack.app.token.<workspaceID>`.
- `SecretRedaction` + `Log` redact tokens/keys from all output.
- Temporary Socket Mode URLs exist only as local variables long enough to create a
  `URLSessionWebSocketTask`; they are never persisted, surfaced, or logged.
- No-egress checks cover Slack and configured LLM clients; Sparkle is restricted to the
  fixed HTTPS GitHub release feed and verifies updates with EdDSA plus Developer ID signing.
- Read-only Slack scopes, user-token, member channels only, no DMs ever.

## Concurrency model

- `SocketModeClient` and `SyncCoordinator` are actors; ingestion/detection run off the main actor.
- View models are `@MainActor @Observable`; they read via GRDB `ValueObservation`.
- GRDB `DatabasePool` gives WAL concurrency; writes are serialized.
- Built in Swift 5 language mode for now (Swift 6 strict concurrency is a later step).

## Testing

XCTest with no real network I/O:
- `HTTPTransport` / `CLIRunner` / `LLMClient` stubs; in-memory `DatabaseQueue`;
  injectable `now`/`makeID`/`sleep` for determinism.
- The app **must not** boot live work under the test host — `AppRoot` early-returns when
  `isRunningUnderTests` (the test bundle is hosted by the app; otherwise live Socket Mode
  would hit live Slack/LLM).

```sh
xcodebuild test -project Slacker.xcodeproj -scheme Slacker -destination 'platform=macOS'
```
